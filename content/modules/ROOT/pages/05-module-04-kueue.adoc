= Module 4: Kueue Concepts
:navtitle: Module 4: Kueue
:toc: macro
:toclevels: 1
:icons: font

Kueue is a Kubernetes-native job queueing system that manages resource quotas, fair sharing, and workload preemption. In this workshop environment, Kueue controls how GPU resources are shared between users with reservations and those using the unreserved pool.

In this module, you'll explore the Kueue resource hierarchy, understand how Cohorts enable resource sharing, and trace a workload through the admission process.

== Learning objectives

By the end of this module, you'll be able to:

* Describe the Kueue resource hierarchy: https://kueue.sigs.k8s.io/docs/concepts/resource_flavor[ResourceFlavor^], https://kueue.sigs.k8s.io/docs/concepts/cluster_queue[ClusterQueue^], https://kueue.sigs.k8s.io/docs/concepts/local_queue[LocalQueue^], https://kueue.sigs.k8s.io/docs/concepts/cohort[Cohort^]
* Explain how the unreserved Cohort enables borrowing and reclaiming
* Submit a workload and trace its admission through Kueue
* Identify when and why workloads are queued vs. admitted

== Kueue resource hierarchy

image::kueue-resource-hierarchy.png[Kueue Resource Hierarchy,link=self,window=blank,width=700]

The workshop cluster configures Kueue resources in a flat Cohort model:

----
Cohort: unreserved
├── ClusterQueue: unreserved        (borrows from cohort, no preemption rights)
├── ClusterQueue: unreserved-priority (can preempt low-priority borrowing workloads)
├── ClusterQueue: user-alice        (protected nominalQuota, can reclaim from cohort)
└── ClusterQueue: user-bob          (protected nominalQuota, can reclaim from cohort)
----

All ClusterQueues share resources through the **unreserved** Cohort. The key difference is in their preemption policies:

[cols="1,1,1,2",options="header"]
|===
| ClusterQueue | nominalQuota | Borrow? | Preempt?

| `unreserved`
| Total minus reserved
| Yes (from cohort)
| No -- workloads here are always preemptable

| `unreserved-priority`
| 0
| Yes (from cohort)
| Yes -- can preempt lower-priority borrowing workloads (see below)

| `user-<name>`
| User's reservation
| No
| Yes -- `reclaimWithinCohort: Any` reclaims its protected quota
|===

=== What is low-priority borrowing?

The `unreserved-priority` ClusterQueue exists for workloads that need to preempt other unreserved work -- without having a reservation. It has `nominalQuota: 0` (no guaranteed resources), so all its resources are borrowed from the Cohort. The difference from the regular `unreserved` queue is its preemption policy:

[source,yaml]
----
preemption:
  borrowWithinCohort:
    policy: LowerPriority
    maxPriorityThreshold: 100
----

This means: "when I am borrowing from the Cohort, I can preempt other borrowing workloads -- but only those with a priority value of 100 or below." A workload using the `high-priority` WorkloadPriorityClass (value `10000`) submitted to `unreserved-priority` can preempt regular unreserved workloads (which have no priority class, defaulting to 0). But it still **cannot** preempt workloads running under a user reservation -- those are protected by `nominalQuota`, not borrowing.

This is useful for batch scheduling scenarios where certain unreserved jobs (e.g., urgent training runs) should jump ahead of background workloads without needing a full reservation.

=== What is protected quota?

When you make a reservation, the app creates Kueue resources that give your workloads **protected quota** -- they cannot be preempted by unreserved workloads.

We have configured reservation guarantees to ensure that user reservations are not preempted by unreserved workloads. This is a design choice to ensure that users can reserve GPUs for their own use and not have them taken away by other users whilst allowing unreserved workloads to run when resources are available.

The Booking App's design is achieved through the use of Kueue's Cohort model:

- the https://kueue.sigs.k8s.io/docs/concepts/cohort[Cohort^] (which is named `unreserved`) initially owns all the resources in the cluster
- all unreserved workloads are borrowing (their https://kueue.sigs.k8s.io/docs/concepts/cluster_queue[ClusterQueue^] has `nominalQuota: 0`).
- when a user makes a reservation, the Cohort is reduced by the amount of the reservation
- a user https://kueue.sigs.k8s.io/docs/concepts/cluster_queue[ClusterQueue^] `user-<name>` is created for the user with the `nominalQuota` matching the reservation
- their `user-<name>` https://kueue.sigs.k8s.io/docs/concepts/cluster_queue[ClusterQueue^] has `reclaimWithinCohort: Any` and `borrowWithinCohort: Never` which means it can preempt any workload borrowing from the cohort but never borrow itself

https://kueue.sigs.k8s.io/docs/concepts/preemption[Preemption^] works because the booking app actively shrinks the Cohort's capacity when quota is allocated. The unreserved CQ has 0 nominal and borrows from the Cohort. When the Cohort shrinks, the unreserved workloads are over their allocation, and the user CQ reclaims.

With this design, we can complement the Kueue Cohort model with other CQ's. One example (not covered in this workshop), is a https://kueue.sigs.k8s.io/docs/concepts/multikueue[Multi-Kueue^] environment. We have a CQ `cluster-queue` where batch workloads are assigned by a HUB Cluster (ACM) to SPOKE GPU enabled clusters. Unreserved workloads can then use the setting `borrowWithinCohort: LowerPriority` to preempt lower priority unreserved workloads - whilst preserving the reservation guarantees for users.

+
* `spec.cohort: unreserved` -- joins the shared Cohort

+
User ClusterQueues have `reclaimWithinCohort: Any` -- they can preempt unreserved workloads to reclaim their protected quota.

== Exercise 3: Trace a workload through admission

. Submit a GPU workload to the unreserved queue:
+
[.copypaste]
[source,bash,subs="attributes"]
----
helm template gpu-deploy oci://quay.io/eformat/gpu-test-workloads -n user-{user-user1} \
  --set kind=Pod \
  --set name=kueue-test \
  --set replicas=1 \
  --set gpuCount=1 \
  --set gpuResourceType=nvidia.com/gpu \
  --set reserved=false \
  | oc -n user-{user-user1} apply -f-
----

. Watch the workload admission:
+
[.copypaste]
[source,bash,subs="attributes"]
----
oc -n user-{user-user1} get workloads -w
----
+
Press `Ctrl+C` after the workload is admitted.

. Check which ClusterQueue admitted the workload:
+
[.copypaste]
[source,bash,subs="attributes"]
----
oc -n user-{user-user1} get workloads -o custom-columns=\
NAME:.metadata.name,\
QUEUE:.spec.queueName,\
ADMISSION:.status.admission.clusterQueue
----
+
Unreserved workloads are admitted by the `unreserved` ClusterQueue.

. View the ClusterQueue usage after admission:
+
[.copypaste]
[source,bash]
----
oc get clusterqueue unreserved -o jsonpath='{.status.flavorsUsage}' | jq .
----

. Clean up:
+
[.copypaste]
[source,bash,subs="attributes"]
----
oc -n user-{user-user1} delete pod kueue-test
----

=== Verify

&#10003; Workload was admitted by the `unreserved` ClusterQueue +
&#10003; ClusterQueue usage reflects the admitted workload +
&#10003; Workload cleaned up

== Exercise 4: Understand the Cohort model

The Cohort is the key to resource sharing. All ClusterQueues in the same Cohort share a single pool of resources:

. View the Cohort resource totals:
+
[.copypaste]
[source,bash]
----
oc get clusterqueues -o custom-columns=\
NAME:.metadata.name,\
COHORT:.spec.cohort \
| grep unreserved
----

. The total resources available in the Cohort equal:
+
`Total GPU capacity - Sum of all user reservations = Unreserved Cohort quota`
+
When a user makes a reservation, the booking app reduces the `unreserved` ClusterQueue's nominalQuota by the reserved amount.

. Check the WorkloadPriorityClasses:
+
[.copypaste]
[source,bash]
----
oc get workloadpriorityclasses
----
+
Higher priority workloads are admitted first and can preempt lower priority ones (within preemption policy rules).

=== Verify

&#10003; Understand that Cohort enables resource sharing across ClusterQueues +
&#10003; Reservation reduces unreserved quota dynamically +
&#10003; WorkloadPriorityClasses control admission order

== Module summary

**What you accomplished:**

* Explored the Kueue resource hierarchy: ResourceFlavors, ClusterQueues, LocalQueues, Cohorts
* Understood the flat Cohort model with protected user quotas and shared unreserved pool
* Traced a workload from submission to Kueue admission
* Learned how reservations dynamically adjust Cohort quotas

**Key takeaways:**

* Kueue provides fair, quota-based GPU scheduling without custom controllers
* The Cohort model allows unused reserved capacity to be borrowed by unreserved workloads
* User ClusterQueues have `reclaimWithinCohort: Any` to preempt unreserved workloads when needed
* ResourceFlavors map Kueue quotas to physical (or simulated) GPU nodes

= Module 2: GPU Booking App
:navtitle: Module 2: GPU Booking
:toc: macro
:toclevels: 1
:icons: font

The https://github.com/rhai-code/gpu-booking-app-plugin[GPU Booking App^] is an OpenShift Console dynamic plugin that provides a calendar interface for reserving GPU resources. It integrates with https://kueue.sigs.k8s.io/docs/overview[Kueue^] to automatically provision per-user ClusterQueues, LocalQueues, and Hardware Profiles based on bookings.

[IMPORTANT]
====
The Booking App was designed and built to provide a simple, intuitive interface for users to reserve GPU resources. It is a community project and actively used to manage GPU resources in real GPU enabled clusters. It is not a Red Hat product and is not officially supported by Red Hat.
====

In this module, you'll use the booking interface to reserve GPU time, observe the Kueue resources it creates, and understand the difference between reserved and consumed bookings.

== Learning objectives

By the end of this module, you'll be able to:

* Navigate the GPU Booking App in the OpenShift Console
* Create reserved bookings for different GPU resource types
* Explain the difference between reserved and consumed bookings
* Inspect the Kueue resources created by the reservation sync

[NOTE]
====
The exercises below use the **NVIDIA GB200 NVL** profile. If you are not using this profile, you will need to switch to it using the GPU Config Plugin. See the xref:08-quickstart.adoc[Quickstart module] for more details. You may use other profiles if you wish - however you will need to adjust the commands accordingly.
====

== How the booking app works

image::booking-sync-lifecycle.png[Booking Sync Lifecycle,link=self,window=blank,width=700]

The booking app has three sync loops that run continuously:

[cols="1,1,3"]
|===
| Loop | Interval | What it does

| **Kueue Sync**
| 30 seconds
| Polls LocalQueues for active workloads and creates _consumed_ bookings reflecting actual GPU usage

| **Reservation Sync**
| 10 minutes
| Reads _reserved_ bookings for today/tomorrow and creates per-user ClusterQueues, LocalQueues, and HardwareProfiles

| **Expiration Cleaner**
| 10 minutes
| Gracefully drains and removes expired ClusterQueues using a two-phase approach
|===

When you make a reservation, the app creates Kueue resources that give your workloads **protected quota** -- they cannot be preempted by unreserved workloads.

We have configured reservation guarantees to ensure that user reservations are not preempted by unreserved workloads. This is a design choice to ensure that users can reserve GPUs for their own use and not have them taken away by other users whilst allowing unreserved workloads to run when resources are available.

Other designs are possible, the Booking App is one of many possible implementations of the Kueue Cohort model. We cover the Kueue Cohort model in more detail in Module 4.

+

+
[source,console,role=execute,subs=attributes+]

----
+

+
[source,console,role=execute,subs=attributes+]

----
+

+
[source,console,role=execute,subs=attributes+]

----
+
You should see a new LocalQueue called `reserved` pointing to your user's ClusterQueue.

== Exercise 3: Understand reserved vs. consumed bookings

There are two types of bookings in the system:

**Reserved bookings** are created when you explicitly book GPU time through the calendar. They:

* Create per-user Kueue resources (ClusterQueue, LocalQueue, HardwareProfile)
* Set `nominalQuota` that protects your allocation from preemption
* Reduce the shared `unreserved` Cohort by the reserved amount

**Consumed bookings** are automatically created by the Kueue Sync loop. They:

* Reflect actual GPU usage detected from active LocalQueue workloads
* Appear as a different color in the calendar
* Are auto-generated with deterministic IDs so the sync can reconcile additions/removals
* Can be overridden by a new reservation (consumed workloads get preempted)

Let's see this in practice. You reserved 4 GPUs in Exercise 2, but now you'll deploy a workload requesting **6** -- more than your reservation. This is allowed, and understanding _why_ is key to understanding the Kueue Cohort model.

[IMPORTANT]
====
**Acting as: `{user-user1}`**
====

. Deploy a pod requesting **6** full GPUs to the reserved queue:
+
[source,console,role=execute,subs=attributes+]
----
helm template gpu-deploy oci://quay.io/eformat/gpu-test-workloads -n user-{user-user1} \
  --set kind=Pod \
  --set name=gpu-test \
  --set replicas=1 \
  --set gpuCount=6 \
  --set gpuResourceType=nvidia.com/gpu \
  --set reserved=true \
  | oc -n user-{user-user1} apply -f-
----

. Wait for the pod to start:
+
[source,console,role=execute,subs=attributes+]
----
oc -n user-{user-user1} wait pod/gpu-test --for=condition=Ready --timeout=60s
----
+
You can also see the workload in the Bookings dashboard.
+
image::consumed-6-gpu.png[6 GPUs Consumed,link=self,window=blank,width=700]
+
You should see 4 **red** slots for the reserved GPUs and 2 **green** slots for the extra consumed GPUs, totalling 6. The Booking App refreshes the screen every 10 seconds, or use the refresh button in the top right.

=== Why 6 GPUs when only 4 are reserved?

Your ClusterQueue `user-{user-user1}` has `nominalQuota: 4` -- that's your guaranteed, protected allocation. But it also belongs to the `unreserved` Cohort, and Kueue allows ClusterQueues to **borrow** idle resources from their Cohort beyond their nominalQuota.

Inspect the ClusterQueue status to see this in action:

[source,console,role=execute,subs=attributes+]
----
oc get cq user-{user-user1} -o jsonpath='{range .status.flavorsUsage[0].resources[*]}{.name}{"\t"}total={.total}{"\t"}borrowed={.borrowed}{"\n"}{end}'
----

You should see output like:

[source,text]
----
cpu          total=0    borrowed=0
memory       total=0    borrowed=0
nvidia.com/gpu   total=6    borrowed=2
----

Kueue reports `total=6` and `borrowed=2` for `nvidia.com/gpu`. The breakdown:

* **4 GPUs** come from your `nominalQuota` -- these are your reserved allocation
* **2 GPUs** are **borrowed** from the Cohort's idle capacity

The Cohort `unreserved` owns the remaining 4 GPUs (8 total minus your 4 reserved). Since no unreserved workloads are running, those 4 GPUs are idle -- and your ClusterQueue borrows 2 of them.

=== What protects what?

The preemption policy on your ClusterQueue controls the difference between reserved and borrowed GPUs:

[cols="1,1,2",options="header"]
|===
| GPU type | Calendar color | Preemption behavior

| **Reserved** (nominalQuota)
| Red
| Fully protected. Your CQ has `reclaimWithinCohort: Any` -- it can preempt any unreserved workload to reclaim these GPUs if needed.

| **Borrowed** (from Cohort)
| Green
| Best-effort. Your CQ has `borrowWithinCohort.policy: Never` -- it cannot preempt anyone to get borrowed GPUs. They must be genuinely idle. If another user reserves those GPUs, your borrowed allocation is preempted first.
|===

This is why the booking app shows two colors: **red** means guaranteed capacity backed by `nominalQuota`, and **green** means your workload is consuming more than its reservation by borrowing idle Cohort resources.

TIP: Borrowed GPUs let you burst beyond your reservation when the cluster has spare capacity, but they come with no guarantee. If you need all 6 GPUs to be protected, increase your reservation to 6 in the booking calendar using the **Override** button.

== Module summary

**What you accomplished:**

* Navigated the GPU Booking App in the OpenShift Console
* Created a GPU reservation and observed the Kueue resources it provisions
* Understood the difference between reserved and consumed bookings
* Explained why workloads can consume more GPUs than reserved through Cohort borrowing

**Key takeaways:**

* The booking app bridges the gap between calendar-based GPU scheduling and Kubernetes resource management
* Reservations create per-user ClusterQueues with protected `nominalQuota`
* Consumed bookings are auto-detected from actual workload usage via Kueue sync
* Workloads can borrow idle Cohort resources beyond their reservation, but borrowed GPUs are not protected from preemption

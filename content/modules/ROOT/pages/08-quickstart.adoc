= Quickstart: End-to-End GPU Booking Demo
:navtitle: Quickstart
:toc: macro
:toclevels: 1
:icons: font

This quickstart exercise is a complete GPU booking workflow end to end. It is self contained and does not require you to have completed any of the modules so is suitable for a standalone walkthrough or demo.

You will configure a GPU profile as an administrator, deploy an unreserved workload as one user, then reserve GPUs and launch a workbench as a second user -- triggering preemption and observing how Kueue manages resource contention automatically.

The follow on Modules explore the individual components in more detail.

== Learning objectives

By the end of this quickstart, you'll be able to:

* Configure and deploy a GPU profile using the GPU Config console plugin
* Deploy unreserved GPU workloads via Helm
* Create a GPU reservation through the booking app
* Launch an RHOAI workbench using a reserved Hardware Profile
* Observe Kueue preemption when reserved workloads claim resources from unreserved ones
* Clean up all resources

== Personas in this quickstart

This walkthrough uses three user identities:

[cols="1,1,3",options="header"]
|===
| User | Role | Actions

| `{user-admin}`
| Cluster admin
| Configures the GPU profile, discovers GPUs, sets up the environment

| `{user-user1}`
| Regular user
| Deploys an unreserved workload consuming all GPUs

| `{user-2}`
| Regular user
| Reserves a GPU, creates an RHOAI workbench, triggers preemption of {user-user1}'s workload
|===

NOTE: This quickstart is using the **NVIDIA GB200 NVL** profile. You may use other profiles if you wish - however you will need to adjust the commands accordingly. Don't worry if the images in this quickstart do not match exactly your GPU profile. The number of GPUs may be slightly different in the images depending on the profile you deploy and number of nodes in the cluster. Remember the GPUs are all simulated in this workshop!

== Exercise 2: Deploy an unreserved workload ({user-user1})

[IMPORTANT]
====
**Acting as: `{user-user1}`** +
Log in to the OpenShift CLI:

[source,console,role=execute,subs=attributes+]
----
oc login --insecure-skip-tls-verify $(oc whoami --show-server) -u {user-user1} -p {password}
----

You should see `Login successful` and the list of accessible projects.
====

=== Deploy workload

. Deploy a workload requesting 8 full GPUs to the unreserved cluster queue:
+
[source,console,role=execute,subs=attributes+]
----
helm template gpu-deploy oci://quay.io/eformat/gpu-test-workloads -n user-{user-user1} \
  --set kind=Deployment \
  --set name=gpu-fake-gpu \
  --set replicas=1 \
  --set gpuCount=8 \
  --set gpuResourceType=nvidia.com/gpu \
  --set reserved=false \
  | oc -n user-{user-user1} apply -f-
----
+
Expected output:
+
[source,text]
----
deployment.apps/gpu-fake-gpu created
----
+
This renders a Helm chart locally and pipes the resulting Deployment manifest into `oc apply`. The workload requests 8 full GPUs and targets the `unreserved` ClusterQueue -- meaning it has no reservation and can be preempted later.

. Wait for the deployment to be ready:
+
[source,console,role=execute,subs=attributes+]
----
oc -n user-{user-user1} rollout status deployment/gpu-fake-gpu --timeout=120s
----
+
Expected output:
+
[source,text]
----
deployment "gpu-fake-gpu" successfully rolled out
----

=== Verify GPU allocation

. Check that the pod can see the GPUs:
+
[source,console,role=execute,subs=attributes+]
----
oc -n user-{user-user1} exec $(oc -n user-{user-user1} get pod -l app=gpu-fake-gpu -o name) -- nvidia-smi
----
+
You should see 8 NVIDIA GB200 NVL GPUs listed in the `nvidia-smi` output table. Each row represents one simulated GPU with its memory usage and utilization.

. Check the Kueue resources:
+
[source,console,role=execute,subs=attributes+]
----
oc get lq,cq,workload -A
----
+
Expected output (abridged):
+
[source,text,subs="attributes"]
----
NAMESPACE    NAME                                        CLUSTERQUEUE   PENDING   ADMITTED
{user-user1}   localqueue.kueue.x-k8s.io/unreserved       unreserved     0         1

NAMESPACE   NAME                                             COHORT       PENDING
            clusterqueue.kueue.x-k8s.io/unreserved           unreserved   0

NAMESPACE    NAME                                                        QUEUE        ADMITTED   AGE
{user-user1}   workload.kueue.x-k8s.io/pod-gpu-fake-gpu-...              unreserved   True       30s
----
+
The key things to observe:
+
* **LocalQueue** `unreserved` in `{user-user1}` has 1 admitted workload -- this is {user-user1}'s deployment.
* **ClusterQueue** `unreserved` belongs to the `unreserved` cohort, which shares GPU resources with reserved queues.
* **Workload** shows `ADMITTED=True` -- Kueue has granted this workload access to GPU resources.

. Open the **GPU Booking** > **Bookings** page in the OpenShift Console to see all 8 full GPUs consumed:

image::qs-user1-workload-running.png[GPU Booking showing 8 GPUs consumed by {user-user1},link=self,window=blank,width=700]

=== Verify

&#10003; Deployment `gpu-fake-gpu` is running in `{user-user1}` namespace +
&#10003; `nvidia-smi` shows 8 GPUs allocated +
&#10003; GPU Booking shows 8/8 full GPUs consumed +
&#10003; RHOAI dashboard is accessible

== Exercise 3: Reserve a GPU and create a workbench ({user-2})

[IMPORTANT]
====
**Acting as: `{user-2}`** +
Log in to the OpenShift Console at `{openshift_console_url}` with username `{user-2}` and password `{password}`.
====

=== Create a GPU booking

. Navigate to **GPU Booking** > **Bookings** in the console menu.

. Scroll down to the calendar and click on **today's date**.

. In the slot grid below the calendar, find the **NVIDIA GB200 NVL Full GPU** row. All 8 units show `{user-user1}` consumed.

. Click **Override** on one of the consumed slots to create a reservation that will preempt the unreserved workload.

image::qs-create-booking.png[GPU Booking calendar with slot grid,link=self,window=blank,width=700]

The slot grid now shows your reservation alongside {user-user1}'s consumed workloads:

image::qs-booking-confirmed.png[Slot grid showing {user-2} reservation and {user-user1} consumed slots,link=self,window=blank,width=700]

image::qs-booking-confirmed-2.png[GPU Booking grid showing {user-2} reservation and {user-user1} consumed slots,link=self,window=blank,width=700]

=== Create an RHOAI workbench

. Open the Red Hat OpenShift AI dashboard:
+
`{openshift_ai_url}`
+
Log in as `{user-2}` with password `{password}`.

. Navigate to **Projects** and select the **User {user-2} Project**.

. Click **Create a workbench** and fill in:
+
* **Name:** `fake-gpu-workbench`
* **Image selection:** Training | Jupyter | PyTorch | CUDA | Python
* **Hardware profile:** Reserved GPU (Project-scoped)

image::qs-workbench-create.png[RHOAI workbench creation form with Reserved GPU hardware profile,link=self,window=blank,width=700]

. Click **Create workbench** and wait for it to reach **Ready** status.

image::qs-workbench-running.png[RHOAI workbench in Ready state,link=self,window=blank,width=700]

=== Verify Kueue state

. Log in to the CLI as `{user-2}`:
+
[source,console,role=execute,subs=attributes+]
----
oc login --insecure-skip-tls-verify $(oc whoami --show-server) -u {user-2} -p {password}
----

. Check the Kueue resources:
+
[source,console,role=execute,subs=attributes+]
----
oc get lq,cq,workload -A
----
+
Expected output (abridged):
+
[source,text,subs="attributes"]
----
NAMESPACE    NAME                                        CLUSTERQUEUE   PENDING   ADMITTED
{user-user1}   localqueue.kueue.x-k8s.io/unreserved       unreserved     2         0
{user-user2}   localqueue.kueue.x-k8s.io/reserved         {user-user2}     0         1

NAMESPACE   NAME                                             COHORT       PENDING
            clusterqueue.kueue.x-k8s.io/unreserved           unreserved   2
            clusterqueue.kueue.x-k8s.io/{user-user2}           unreserved   0

NAMESPACE    NAME                                                        QUEUE        RESERVED IN   ADMITTED   AGE
{user-user1}   workload.kueue.x-k8s.io/pod-gpu-fake-gpu-...-f17e3        unreserved                              16s
{user-user1}   workload.kueue.x-k8s.io/pod-gpu-fake-gpu-...-0fae4        unreserved                  False      13m
{user-user2}   workload.kueue.x-k8s.io/statefulset-fake-gpu-workbench-.. reserved     {user-user2}    True       16s
----
+
This output shows preemption in action:
+
* **{user-user1}'s workloads** are now `ADMITTED=False` or have no admission status. Kueue evicted them to free the GPU for {user-2}'s reservation. The `unreserved` queue shows 2 pending workloads that are waiting for resources.
* **{user-2}'s workload** shows `ADMITTED=True` and `RESERVED IN={user-user2}`. The booking app created a dedicated ClusterQueue (`{user-user2}`) for the reservation, and the workbench workload was admitted there with priority over the unreserved workloads.
* The `{user-user2}` ClusterQueue belongs to the same `unreserved` cohort -- this is how Kueue's `reclaimWithinCohort` policy allows reserved workloads to reclaim resources from unreserved ones.

=== Verify

&#10003; GPU booking created for {user-2} +
&#10003; Workbench `fake-gpu-workbench` is running in RHOAI +
&#10003; {user-2}'s ClusterQueue and LocalQueue are visible in Kueue

== Exercise 4: Observe preemption

When {user-2}'s reserved workload starts, Kueue automatically evicts {user-user1}'s unreserved pod to free the GPU. You can observe this in both the CLI and the GPU Booking app.

=== {user-user1}: Watch pod preemption

[IMPORTANT]
====
**Acting as: `{user-user1}`** +
Switch to the {user-user1} CLI context:

[source,console,role=execute,subs=attributes+]
----
oc login --insecure-skip-tls-verify $(oc whoami --show-server) -u {user-user1} -p {password}
----
====

. Check the pods in {user-user1}'s namespace:
+
[source,console,role=execute,subs=attributes+]
----
oc -n user-{user-user1} get pods
----
+
Expected output:
+
[source,text]
----
NAME                            READY   STATUS            RESTARTS   AGE
gpu-fake-gpu-54959878d8-g88dn   0/1     SchedulingGated   0          15m
----
+
{user-user1}'s pod shows `SchedulingGated` -- Kueue has evicted it to reclaim the GPU for {user-2}'s reservation. The pod is not terminated; it is held in a gated state and will automatically resume once the GPU becomes available again.

. Open the **GPU Booking** > **Bookings** page to see the preemption reflected in the usage overview. {user-2}'s reservation (red) now holds 1 GPU:

image::qs-preemption-user1.png[GPU Booking showing preemption -- {user-2} reservation active while {user-user1} is evicted,link=self,window=blank,width=700]

=== {user-2}: Delete workbench and observe recovery

[IMPORTANT]
====
**Acting as: `{user-2}`** +
Return to the RHOAI dashboard as {user-2}.
====

. In RHOAI, navigate to **Projects** > **User {user-2} Project** > **Workbenches**.

. Delete the `fake-gpu-workbench` workbench using the kebab menu (three dots).

. Once the workbench is deleted, {user-2}'s workload releases the GPU. {user-user1}'s pod will automatically recover and return to `Running` status.

=== {user-user1}: Pod recovers

[IMPORTANT]
====
**Acting as: `{user-user1}`**
====

. Verify {user-user1}'s pod is running again:
+
[source,console,role=execute,subs=attributes+]
----
oc -n user-{user-user1} get pods
----
+
Expected output:
+
[source,text]
----
NAME                            READY   STATUS    RESTARTS   AGE
gpu-fake-gpu-54959878d8-4gknt   1/1     Running   0          2m
----
+
Once {user-2}'s workbench released the GPU, Kueue automatically re-admitted {user-user1}'s workload. The pod transitions from `SchedulingGated` back to `Running` without any manual intervention.

=== {user-2}: Cancel booking

[IMPORTANT]
====
**Acting as: `{user-2}`** +
Return to the OpenShift Console as {user-2}.
====

. Navigate to **GPU Booking** > **Bookings** and click **My Bookings**.

. Click **Cancel All** to remove the GPU reservation, then click **Confirm**.

image::qs-cancel-booking.png[GPU Booking My Bookings with Cancel All button,link=self,window=blank,width=700]

. Verify that the Kueue resources for {user-2} are removed:
+
[source,console,role=execute,subs=attributes+]
----
oc -n user-{user-user2} get cq,lq,hardwareprofile
----
+
Expected output:
+
[source,text,subs="attributes"]
----
No "reserved" resources found in {user-user2} namespace.
----
+
When a booking is canceled, the GPU Booking app automatically cleans up all Kueue resources it created: the per-user ClusterQueue (`{user-user2}`), the `reserved` LocalQueue, and any HardwareProfiles. This ensures no stale reservation state remains on the cluster.

=== Verify

&#10003; {user-user1}'s pod was preempted (`SchedulingGated`) when {user-2}'s reserved workload started +
&#10003; {user-user1}'s pod recovered to `Running` after {user-2} deleted the workbench +
&#10003; {user-2}'s ClusterQueue, LocalQueue, and HardwareProfiles removed after canceling the booking

== Module summary

**What you accomplished:**

* Configured the NVIDIA GB200 NVL GPU profile as a cluster administrator
* Deployed an unreserved GPU workload consuming all available GPUs
* Created a GPU reservation and launched an RHOAI workbench with a reserved Hardware Profile
* Observed Kueue preemption in action -- reserved workloads automatically evict unreserved ones
* Cleaned up all resources

**Key takeaways:**

* The GPU Config plugin lets administrators select and deploy GPU profiles with a single click
* The GPU Booking app creates Kueue resources (ClusterQueues, LocalQueues, HardwareProfiles) from calendar reservations
* Reserved workloads always take priority over unreserved ones through Kueue's `reclaimWithinCohort` policy
* Preemption is fully automatic -- users only interact with the booking calendar and RHOAI
* When a reservation is canceled, all associated Kueue resources are cleaned up automatically
